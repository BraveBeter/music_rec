这份设计文档专为具备扎实全栈开发与算法基础的工程化项目打造，结构严谨，特别强化了多模态融合与新型生成式推荐等扩展探索目标的系统级支撑。

# 音乐推荐系统软件工程设计文档

## 1. 需求分析

### 1.1 系统背景

本课题围绕音乐推荐系统展开研究，旨在对比多种主流推荐技术路线，构建兼顾准确性、泛化能力与用户体验的个性化音乐推荐模型 。系统不仅需要实现基础的推荐业务闭环，还需支持前沿算法的验证与对比。

### 1.2 系统目标

- 在统一数据集上完成实现、训练、评测与误差分析 。

- 基于清晰的推荐技术路线构建一个可交互的音乐推荐系统原型 。

- 适度引入多模态特征（音频 embedding / 标签特征）作为扩展探索 。

- 尝试加入轻量级混合推荐策略或模型蒸馏 。

- 对业界新型推荐范式（如生成式推荐 / OneRec）进行调研性实验或对比分析 。

### 1.3 用户角色

- **普通用户**：浏览推荐音乐、播放歌曲、进行评分与收藏、搜索特定曲目。

- **系统管理员/算法工程师**：管理曲库数据、监控模型运行状态、查看离线实验多指标评测结果（Precision、Recall、NDCG、覆盖率等）、管理Docker容器部署 。

### 1.4 功能需求

- **数据预处理与特征工程**：完成公开数据集（如 Million Song Dataset / LastFM）的清洗、特征构建、行为序列建模及负采样 。

- **个性化推荐展示**：向用户展示基于离线训练和在线 Top-K 推荐生成的个性化歌单 。

- **多模态融合扩展**：在推荐结果中体现轻量多模态融合（音频 embedding + 行为序列）的优势 。

- **系统交互闭环**：捕获用户的短序列/长序列播放行为，用于后续的序列推荐模型（如 BERT4Rec 或 SASRec）训练 。

### 1.5 非功能需求

- **性能**：在线推理响应延迟需满足实时交互要求，离线训练需支持海量数据批处理。

- **可扩展性**：算法模块需具备高度可插拔性，方便对比工业推荐系统“三阶段架构”与本研究方案的差异 。

- **可部署性**：系统需支持 Docker 部署一键启动，保障环境一致性 。

### 1.6 用例分析

- **获取个性化推荐**：用户登录系统后，前端向后端请求推荐列表，后端调用推荐算法模块（如 DeepFM 或 SASRec 模型推理）返回 Top-K 歌曲。

- **冷启动探索**：针对新用户或新歌曲，系统利用多模态特征（如音频特征、风格标签）进行内容推荐，缓解稀疏场景下的冷启动问题 。

## 2. 系统总体设计

### 2.1 系统架构设计

系统采用前后端分离与算法微服务解耦的架构，核心分为三层：表现层、业务逻辑层与模型推理层。

### 2.2 系统整体结构说明

- **前端表现层**：采用 Vue3 / Next.js 构建响应式 Web 界面，负责与用户直接交互，展示音乐列表与播放控制 。

- **后端业务层**：使用 Python + Flask / FastAPI 构建，负责处理 HTTP 请求、鉴权、数据存取与日志记录 。选择 Python 有利于与底层机器学习生态无缝对接。

- **推荐算法层**：独立的服务模块，包含 Baseline（ItemCF / UserCF & 矩阵分解 SVD）、特征建模（FM / DeepFM）与序列推荐（BERT4Rec / SASRec）等多种管线 。支持对生成式推荐 / OneRec 进行小规模实验验证 。

### 2.3 技术栈建议

- **前端**：Vue3 (组合式 API 提供优秀的逻辑复用能力) / TypeScript / Pinia

- **后端**：FastAPI (异步支持，极佳的 API 响应性能) / Python 3.10+

- **推荐算法**：PyTorch / Transformers (用于 BERT4Rec/生成式模型) / FAISS (用于海量音频 Embedding 的高维向量相似度检索)

- **数据库**：MySQL (核心业务数据) + Redis (缓存实时行为序列与离线 Top-K 结果)

### 2.4 系统运行流程

1. 离线阶段：定时拉取交互日志，进行数据清洗与负采样 。分别训练协同过滤、DeepFM 及序列推荐模型，将生成的音频 embedding 与物品隐向量导入向量数据库。

2. 在线阶段：用户发起请求，FastAPI 后端获取用户画像与实时行为序列。

3. 混合路由：根据请求场景（如是否冷启动 ），选择轻量多模态融合策略或特定的召回/排序模型进行在线 Top-K 推理 。
